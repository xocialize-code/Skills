# Check-in

Check-in is the server-side endpoint a device hits for **identity and lifecycle** events â€” not for command execution. Check-in messages are device-initiated and arrive over HTTPS to the URL specified in the MDM payload's `CheckInURL` (or `ServerURL` if `CheckInURL` is absent).

Apple doc: `https://developer.apple.com/documentation/devicemanagement/check-in`
Schemas: `https://github.com/apple/device-management/tree/release/mdm/checkin`

## Check-in messages you'll receive

Each is a property-list POST. The `MessageType` field discriminates.

| MessageType | When it fires | Server should |
|---|---|---|
| `Authenticate` | First contact after enrollment install | Create the device record; verify `Topic` matches your MDM cert UID |
| `TokenUpdate` | After `Authenticate`, on push cert change, on user logout for User Enrollment, periodically | Persist latest `Token` + `PushMagic` keyed by `UDID` (or `EnrollmentID`) |
| `CheckOut` | User removes MDM profile (only fires if `CheckOutWhenRemoved: true` in MDM payload) | Mark device unenrolled; clean up |
| `UserAuthenticate` | Network user (macOS shared/mobile accounts) wants directory binding info | Return optional plist with auth challenge response |
| `GetToken` | Device needs an opaque bootstrap-style token for a service | Return the requested token type |
| `SetBootstrapToken` | macOS device escrows its bootstrap token to you | Store securely; needed to authorize software updates and admin operations |
| `GetBootstrapToken` | Device asks for its escrowed token back (e.g., post-erase) | Return the stored token if device is still entitled |
| `DeclarativeManagement` | Initial DDM activation; sent by server to device then device follows up | See `ddm.md` |

The `DeclarativeManagement` flow is a check-in message in the sense that DDM rides the same check-in transport, but it's also how DDM is initially activated by the server. Treated separately in `ddm.md`.

## Authenticate body â€” key fields

```
MessageType: "Authenticate"
UDID:        "00008140-001234567890ABCD"      // macs use the hardware UUID
Topic:       "com.apple.mgmt.External.UUID"   // must match your MDM push cert
DeviceName:  "Dustin's MacBook Pro"
Model, ModelName, OSVersion, BuildVersion, ProductName, SerialNumber
Challenge:    [opaque data from EnrollmentProfile's `Challenge` key, if set]
```

For User Enrollment / Account-Driven enrollment, `UDID` is absent and `EnrollmentUserID` + `EnrollmentID` are present instead. Code defensively for both shapes.

**Validate `Topic` against your push certificate UID.** If they don't match, the device enrolled against a different MDM service identity and you can't push to it. Reject the Authenticate or store with a flag.

## TokenUpdate body â€” what to persist

```
MessageType:    "TokenUpdate"
Topic:          "com.apple.mgmt.External.UUID"
UDID / EnrollmentID
PushMagic:      "ABCDEF12-3456-7890-..."        // opaque, generated by device
Token:          <Data: 32-byte APNs device token>
UnlockToken:    <Data>                          // present on macOS, used for FileVault unlock; treat as secret
AwaitingConfiguration: true | false             // device is in Setup Assistant, send config now
NotOnConsole:   true | false                    // macOS: TokenUpdate from a logged-out state
UserID, UserShortName, UserLongName             // present for per-user check-ins on macOS
```

Store `Token` and `PushMagic` together â€” you need both to wake the device. They change over time; always trust the latest `TokenUpdate`.

**`AwaitingConfiguration: true`** is the cue to send the device's setup commands *now* â€” profile installs, app installs, etc. The Setup Assistant is paused waiting for you. When you've queued everything for setup, send the `DeviceConfigured` command to release Setup Assistant. Failing to release leaves the device stuck on a "Remote Management" screen.

## CheckOut

Only fires if your MDM payload was installed with `CheckOutWhenRemoved: true` (top-level key in the MDM payload). Otherwise removal is silent.

```
MessageType: "CheckOut"
Topic, UDID / EnrollmentID
```

On CheckOut: mark the device record unenrolled, but keep historical records for audit. The device's identity cert is now untrusted by the server (and the device has lost the MDM payload anyway). Do not delete the row immediately â€” you may want to reconcile against ABM/ASM later.

## The push identity triad refresher

Re-stating because TokenUpdate is the moment all three meet:

- **Topic** = your MDM push cert's `UID`. Same for every device served by this cert.
- **PushMagic** = per-device wake key. Goes in the APNs payload body as `{"mdm": "[PushMagic]"}`.
- **Token** = per-device APNs device token. Goes in the APNs HTTP/2 `:path` as `/3/device/[hex-encoded-Token]`.

To wake device:
```
POST https://api.push.apple.com/3/device/[hex(Token)]
Headers:
  apns-topic:    [Topic]
  apns-push-type: mdm
  authorization: bearer [provider auth token]
Body:
  {"mdm": "[PushMagic]"}
```

Use the **provider auth token** flow (JWT signed with an APNs Auth Key) rather than per-connection TLS client cert, unless you have a reason to use the older path.

## UserAuthenticate (macOS network accounts)

Used when a network user logs into a macOS device with a directory-bound account that needs MDM-side authentication. Two-phase challenge/response. Most modern MDMs don't implement this â€” Apple has been deprecating the directory-bound network user flow. Only worry about it if you have a specific need.

## SetBootstrapToken / GetBootstrapToken (macOS)

The **bootstrap token** is a macOS-specific secret the device generates and escrows to your MDM so the MDM can authorize sensitive operations (software updates, kext approvals, login window changes) without prompting the user.

- `SetBootstrapToken`: device sends the token to you. Store it encrypted, scoped to the device. Treat with the same care as a password.
- `GetBootstrapToken`: device requests it back. Verify the device is still entitled (still enrolled, same UDID, etc.) before returning.

A device that has not escrowed its bootstrap token cannot authorize certain operations even if your MDM is fully trusted. After enrollment, you typically want to trigger bootstrap token escrow ASAP.

## Gotchas

- **MDM payload's `CheckInURL`** is optional. If absent, check-in goes to `ServerURL`. Many homebrew MDM setups omit `CheckInURL` and tunnel everything through `ServerURL`; that works, but if you support both, your server needs to dispatch by Content-Type or path.
- **MDM payload's `SignMessage: true`** means the device CMS-signs all check-in and command-poll bodies with its identity certificate. You must verify the signature using the cert chain from enrollment. If your server skips verification, anyone with the URL can spoof a device â€” and it's been seen in the wild.
- **TokenUpdate frequency varies.** Treat any stale-feeling token as suspect and re-push to verify; failed push is a sign the device has rolled tokens or unenrolled silently.
- **User Enrollment uses `EnrollmentID`, not `UDID`.** Your storage layer needs a polymorphic device key. Don't assume `UDID` is non-null.
- **`AwaitingConfiguration`** appears on the TokenUpdate immediately following Authenticate in an ADE flow. If you miss this window, the device proceeds without your initial configuration.
